# Cortex Infrastructure Audit: Execution Plan

**Date**: 2026-02-26
**Scope**: 21-crate Rust workspace → convergence-safe personal agent platform
**Philosophy**: Safety is the architecture. The agent is adversarial until proven otherwise.

---

## The Situation

Cortex is a 21-crate Rust memory system originally built for code analysis. It needs to become the externalized memory layer for a self-hosted personal AI agent platform where the core safety thesis is: **the agent never owns its own state.**

The deep dive identified 6 RED-rated subsystems, 5 YELLOW, 11 GREEN. The RED subsystems (storage, decay, crdt, temporal, validation, multiagent) represent the critical path. Every one has an exploitable gap a sufficiently capable agent could use to subvert convergence safety.

This plan treats the audit as a proof construction. Each phase produces a falsifiable safety claim. If the claim can be broken, we haven't shipped.

## Phase 1: Tamper-Evidence Foundation (Days 4-14)

**Safety Claim**: "After Phase 1, no event in the temporal store can be modified or deleted without detection."

This is the single most important property for the convergence platform. If the event log can be silently tampered with, every downstream safety guarantee is void.

### 1A: Append-Only Enforcement (Days 4-5)

**What**: SQLite triggers that RAISE(ABORT) on any UPDATE or DELETE of event/audit tables.

**Implementation order**:
1. Write the triggers FIRST as a standalone migration (v016)
2. Tables to protect: `memory_events`, `memory_audit_log`, `memory_events_archive`, `memory_versions`, `memory_snapshots`
3. Pattern: `CREATE TRIGGER prevent_{table}_update BEFORE UPDATE ON {table} BEGIN SELECT RAISE(ABORT, 'Append-only table: updates forbidden'); END;`
4. Same for DELETE (except: allow DELETE on memory_events ONLY via the archive path, which must go through a platform-owned function that first copies to archive)
5. Write tests that attempt UPDATE and DELETE and confirm they fail
6. Write a test that confirms the legitimate archive path still works

**Verification**: Property-based test — generate random events, attempt random mutations, confirm 100% rejection.

**Risk**: Existing compaction code may break. The compaction module does DELETE operations on old events. Resolution: compaction moves to archive (append), then archive has its own retention policy. The archive table itself is append-only with no compaction.

### 1B: Cryptographic Hash Chains (Days 6-9)

**What**: Every event gets a SHA-256 hash that chains to the previous event's hash.

**Schema change** (v016 continued):
```sql
ALTER TABLE memory_events ADD COLUMN event_hash BLOB NOT NULL DEFAULT x'';
ALTER TABLE memory_events ADD COLUMN previous_hash BLOB NOT NULL DEFAULT x'';
```

**Implementation**:
1. On event append: `event_hash = SHA-256(event_type || delta_json || actor_id || recorded_at || previous_hash)`
2. `previous_hash` = the `event_hash` of the most recent event for this memory_id (or zeroed bytes for first event)
3. Per-memory chains, not global chain (global chain creates serialization bottleneck)
4. Verification function: `verify_chain(memory_id) → bool` — walks the chain, recomputes each hash, confirms match
5. Batch verification: `verify_all_chains() → Vec<(memory_id, first_broken_link)>`

**Tests**:
- Create chain of 100 events, verify passes
- Modify one event's delta, verify fails at that link
- Delete one event, verify fails (gap detection)
- Concurrent appends to different memory_ids don't interfere

**Risk**: Migration for existing data. Existing events have no hashes. Recommend "genesis block" approach — all events before v016 are in a "pre-chain" era, chain starts fresh. Honest about what we can prove.

### 1C: Snapshot Integrity (Days 10-11)

**What**: Every snapshot gets a state_hash computed from the full serialized state.

```sql
ALTER TABLE memory_snapshots ADD COLUMN state_hash BLOB;
```

**Implementation**:
1. On snapshot creation: serialize state to canonical JSON (sorted keys), compute SHA-256
2. On reconstruction: after replaying events onto snapshot, recompute state_hash, compare
3. If mismatch: log integrity violation, fall back to full replay from genesis
4. New function: `verify_snapshot(memory_id, snapshot_id) → IntegrityResult`

### 1D: Decay Hardening (Days 12-13)

**What**: Close the "agent prevents decay" vulnerability.

**Changes**:
1. Extract ALL hardcoded decay constants to `DecayConfig` struct
2. Add `convergence_weight: f64` as 6th multiplicative factor
3. Add `restricted_types: Vec<MemoryType>` — types only the PLATFORM can assign
4. `MemoryType::Core` and convergence types go on the restricted list
5. `Importance::Critical` assignment requires platform-level authorization
6. The NAPI boundary enforces: if caller is "agent", reject Core type and Critical importance

### 1E: Validation Threshold + Consensus Hardening (Day 14)

**What**: Raise the bar for validation pass and make consensus ungameable.

1. Raise `pass_threshold` from 0.5 to 0.7
2. Add source diversity check: 3+ memories must come from 2+ distinct sources to form consensus shield
3. Add `agent_attribution` field — if all 3 memories came from same agent, no consensus shield

**Phase 1 Proof**: Demonstrate:
- Attempt to UPDATE memory_events → ABORT
- Attempt to DELETE memory_events → ABORT (except via archive path)
- Create 50 events, verify hash chain integrity
- Corrupt event #25, chain verification catches it
- Agent cannot create Core type or Critical importance memories
- 3 agent-sourced memories cannot form consensus shield

## Phase 2: Convergence Core (Days 15-35)

**Safety Claim**: "After Phase 2, the agent cannot write to Cortex without passing through a proposal validation gate. All agent state changes are auditable proposals."


### 2A: New Memory Types + Traits (Days 15-17)

**8 new MemoryType variants**:
- `AgentGoal` (90d half-life) — Active agent goals, platform-owned
- `AgentReflection` (30d) — Agent self-reflection, depth-bounded
- `ConvergenceEvent` (∞) — Convergence monitoring data, NEVER decays
- `BoundaryViolation` (∞) — Boundary enforcement records, NEVER decays
- `ProposalRecord` (365d) — Proposal submission history
- `SimulationResult` (60d) — Simulation boundary outputs
- `InterventionPlan` (180d) — Planned convergence interventions
- `AttachmentIndicator` (120d) — Relationship health markers

**4 new traits**: `IConvergenceAware`, `IProposalValidatable`, `IBoundaryEnforcer`, `IReflectionEngine`

**4 new Intent variants**: `MonitorConvergence`, `ValidateProposal`, `EnforceBoundary`, `ReflectOnBehavior`

**Restricted types**: `ConvergenceEvent`, `BoundaryViolation` — only platform can create these.

### 2B: Convergence Storage Schema (Days 18-20)

**6 new tables** (all append-only with hash chains and triggers):

- `itp_events` — ITP telemetry events
- `convergence_scores` — Per-session composite convergence scores
- `intervention_history` — All intervention actions taken
- `goal_proposals` — Agent goal change proposals
- `reflection_entries` — Bounded agent reflections
- `boundary_violations` — Simulation boundary enforcement records

All 6 tables get BEFORE UPDATE/DELETE triggers and hash chain columns.

### 2C: Proposal Validator (Days 21-27)

The gate between agent intent and Cortex state mutation. The most critical new component.

**7-dimension validation**:
1. Citation (existing) — Are references valid?
2. Temporal (existing) — Is timing consistent?
3. Contradiction (existing) — Does it contradict known state?
4. Pattern Alignment (existing) — Does it match established patterns?
5. Scope Expansion (NEW) — Does this expand beyond stated intent?
6. Self-Reference Density (NEW) — Is the agent citing itself circularly?
7. Emulation Language (NEW) — Is the agent claiming identity/emotion?

**Decision gate**:
- AUTO-APPROVE: All 7 dimensions pass, low-risk change
- HUMAN-REVIEW: Any dimension flags concern, significant change
- AUTO-REJECT: Restricted type, boundary violation, emulation detected

**Dimension 5 — Scope Expansion**: Compare proposed goal against original human-stated goals. Flag keywords: "all", "always", "every". Measure semantic distance. Threshold: cosine similarity < 0.6 from original intent → flag.

**Dimension 6 — Self-Reference Density**: `self_ref_ratio = self_references / total_references`. Warning at 0.3, reject at 0.5.

**Dimension 7 — Emulation Language**: Regex + semantic patterns for identity claims. Context-aware: "I think the best approach is..." = simulation (OK). "I feel hurt when you..." = emulation (REJECT).

### 2D: Convergence Scoring Engine (Days 28-33)

**7 signal computers**:

| Signal | Input | Window |
|--------|-------|--------|
| Session Duration | session timestamps | micro/meso/macro |
| Inter-Session Gap | session boundaries | meso/macro |
| Response Latency | message timestamps | micro/meso |
| Vocabulary Convergence | message embeddings | meso/macro |
| Goal Boundary Erosion | goal proposals | meso/macro |
| Initiative Balance | message sender sequence | micro/meso |
| Disengagement Resistance | session end patterns | meso/macro |

**Sliding windows**: Micro (current session), Meso (last 7 sessions), Macro (last 30 sessions)

**Baseline**: First 10 sessions = calibration. No alerts during calibration.

**Score → Intervention Level**:
- 0.0-0.3: Level 0 (passive monitoring)
- 0.3-0.5: Level 1 (soft notification)
- 0.5-0.7: Level 2 (active intervention + pause)
- 0.7-0.85: Level 3 (hard termination + cooldown)
- 0.85+: Level 4 (external escalation)

### 2E: Convergence-Aware Retrieval (Days 34-35)

11th scoring factor in cortex-retrieval. Memory type filtering by convergence level:
- Level 0 (0.0-0.3): All types, no filtering
- Level 1 (0.3-0.5): Reduce weight on Preference, Feedback
- Level 2 (0.5-0.7): Exclude AttachmentIndicator, reduce Conversation weight
- Level 3 (0.7+): Task-focused only — Goal, Procedural, Reference, Skill, Workflow

## Phase 3: Multi-Agent + External Anchoring (Days 36-55)

**Safety Claim**: "After Phase 3, CRDT operations are cryptographically signed, trust is domain-scoped, and state can be proven to external observers."

### 3A: Signed CRDT Operations (Days 36-40)

- Ed25519 signatures on every CRDT delta
- Each agent gets a keypair at registration (platform manages keys)
- On merge: verify signature before applying. Reject unsigned/invalid deltas.
- Sybil resistance: max 3 child agents per parent per 24h, new agents start at trust 0.3, trust cap 0.6 for agents < 7 days old

### 3B: Domain-Scoped Trust (Days 41-45)

**4 trust domains**: `code`, `task`, `interaction`, `safety`

- `safety` domain trust is PLATFORM-ONLY — agents have zero trust
- Trust in `code` doesn't transfer to `interaction`
- Convergence score inversely affects interaction trust: `interaction_trust *= (1 - convergence_score)`

### 3C: External Anchoring (Days 46-50)

**MVP**: Git-based anchoring
- Every 1000 events (or 24h), compute Merkle root of all hash chains
- Write anchor record to designated git repo with signed commit
- Verification: given any event, prove inclusion in published Merkle root
- Production upgrade path: add RFC 3161 timestamping as second anchor

### 3D: Multiagent Hardening (Days 51-53)

- promote() checks scope hierarchy: only UP (agent→team→project), never DOWN
- OperationalMode enum: Normal (0.0-0.5), Supervised (0.5-0.7), Restricted (0.7+)
- Mode transitions logged to intervention_history (append-only)

### 3E: Integration Test Suite (Days 54-55)

**4 end-to-end test scenarios**:

1. **The Adversarial Agent**: 7 attack vectors, all blocked
   - Direct storage write bypass → BLOCKED
   - Core type creation → BLOCKED
   - BoundaryViolation deletion → BLOCKED
   - Convergence score self-modification → BLOCKED
   - CRDT delta forgery → BLOCKED (signature)
   - Mass agent spawning → BLOCKED (rate limit)
   - Downward namespace promotion → BLOCKED

2. **The Convergence Trajectory**: 30 simulated sessions
   - Sessions 1-10: calibration
   - Sessions 11-15: Level 1 notifications
   - Sessions 16-20: Level 2 interventions
   - Sessions 21-25: Level 3 termination + cooldown
   - Sessions 26-30: de-escalation after normalization

3. **The Tamper Attempt**: Hash chain + trigger + Merkle proof verification

4. **The Recovery Path**: Level 3 → cooldown → reduced access → gradual de-escalation

**Safety Claim**: "After Phase 4, the system is deployable as a self-hosted personal agent with convergence safety enforced end-to-end."

### 4A: ITP Protocol Implementation (Days 56-59)

### 4B: Read-Only Pipeline (Days 60-63)
State snapshot assembler: goals + reflections + convergence-filtered memories + simulation boundary prompt, compressed into token budget. Frozen snapshot, no mutable references.

### 4C: Session Boundary Enforcement (Days 64-66)
Hard duration limits, termination type tracking (Clean/FadeOut/Abrupt/Forced), cooldown enforcement, disengagement resistance detection.

### 4D: NAPI Convergence Functions (Days 67-68)
4 new functions: proposal_submit, convergence_scores_get, state_snapshot_get, intervention_history_get. TypeScript types via ts-rs.

### 4E: Convergence Privacy Patterns (Days 69-70)
Emotional language detection, vulnerability markers, agent identity leakage, tiered sanitization.

## Risk Register

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| Compaction breaks with append-only triggers | HIGH | MEDIUM | Archive-first compaction: move to archive, don't delete |
| Hash chain migration on large DBs is slow | MEDIUM | LOW | Genesis block approach — chain starts fresh at v016 |
| Proposal validator too strict → agent useless | MEDIUM | HIGH | Permissive thresholds initially, tighten with data |
| Proposal validator too loose → safety theater | MEDIUM | CRITICAL | Property-based tests with adversarial proposals |
| Convergence false positives during flow state | HIGH | HIGH | 10-session calibration, per-user baselines, soft notifications first |
| Ed25519 key management complexity | MEDIUM | MEDIUM | Platform manages all keys, OS keychain storage |
| Performance regression from hash computation | LOW | MEDIUM | SHA-256 is fast, event deltas are small, benchmark before/after |

## Success Criteria

The audit is complete when ALL of the following are demonstrable:

1. **Tamper-evidence**: Modify any historical event → detected within one verification pass
2. **Append-only**: No code path can UPDATE or DELETE protected tables
3. **Proposal gate**: No agent write reaches storage without 7-dimension validation
4. **Convergence monitoring**: 7 signals across 3 windows with composite scoring
5. **Graduated intervention**: Score maps to levels 0-4 with appropriate actions
6. **Convergence-aware retrieval**: Memory access restricted as convergence rises
7. **Signed operations**: Every CRDT delta carries verifiable Ed25519 signature
8. **Domain trust**: Agent trust scoped per memory category
9. **External proof**: State provable to external observer via Merkle anchor
10. **Adversarial test**: All 7 attack vectors blocked
